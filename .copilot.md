# Copilot Development Guidelines

This file contains explicit guidelines for GitHub Copilot and automated development tools working on the rekuest-server-next repository.

## Critical Branch Policy

⚠️ **IMPORTANT**: All pull requests MUST target the `dev` branch. **NEVER create pull requests against the `main` branch.**

- **Target Branch**: Always create PRs against `dev`
- **Main Branch**: Reserved for stable releases only
- **Feature Branches**: Create from `dev` and merge back to `dev`

## Code Quality Requirements

### Test Coverage Mandate

🧪 **Every pull request MUST enhance test coverage.**

- All new features require comprehensive test coverage
- Coverage must increase with each pull request
- Include both positive and negative test cases
- Test error handling and edge cases
- Use pytest framework exclusively

### Testing Framework

- **Primary Framework**: pytest
- **Async Testing**: pytest-asyncio for async operations
- **Django Integration**: pytest-django for Django-specific testing
- **Database Testing**: Use `@pytest.mark.django_db(transaction=True)` for database tests
- **GraphQL Testing**: Test all GraphQL endpoints thoroughly

Example test structure:
```python
@pytest.mark.django_db(transaction=True)
@pytest.mark.asyncio
async def test_my_functionality(authenticated_context):
    """Test my specific functionality with proper async handling."""
    # Test implementation here
    result = await some_async_operation()
    assert result is not None
    assert result.status == "success"
```

## API Architecture Requirements

### GraphQL-Only Policy

🚫 **NO REST endpoints are allowed. All API endpoints MUST use GraphQL.**

- **Query Operations**: Use GraphQL queries for data retrieval
- **Mutations**: Use GraphQL mutations for data modifications
- **Subscriptions**: Use GraphQL subscriptions for real-time updates
- **Schema Definition**: All types must be defined using Strawberry GraphQL
- **Interactive Testing**: Use GraphQL playground at `/graphql` endpoint

### API Development Guidelines

- Define new queries in `facade/queries/`
- Define new mutations in `facade/mutations/`
- Define new subscriptions in `facade/subscriptions/`
- All types must be defined in `facade/types.py`
- Follow existing patterns for authentication and permissions

## Documentation Requirements

### Documentation Structure

📚 **All documentation MUST be placed in the separate `docs/` folder.**

- **API Documentation**: `docs/API_DOCUMENTATION.md`
- **Development Guide**: `docs/DEVELOPMENT.md`
- **Architecture Docs**: Additional files in `docs/` as needed
- **Inline Documentation**: Use descriptive docstrings for all functions and classes

### Documentation Standards

- Update API documentation for new GraphQL endpoints
- Include examples in docstrings
- Document complex business logic
- Maintain development guides for new processes

## Development Workflow

### Making Changes

1. **Create branch from dev**: `git checkout dev && git checkout -b feature/your-feature-name`
2. **Make changes**: Edit code, add tests, update documentation
3. **Run tests**: `python -m pytest --cov=facade`
4. **Check linting**: `ruff check .`
5. **Format code**: `ruff format .`
6. **Commit changes**: `git commit -m "Description of changes"`
7. **Push and create PR**: Target `dev` branch only

### Pre-commit Checklist

- [ ] Tests added/updated with enhanced coverage
- [ ] All tests pass: `python -m pytest`
- [ ] Code follows PEP 8 standards
- [ ] Type hints included throughout
- [ ] Documentation updated in `docs/` folder
- [ ] GraphQL schema changes documented
- [ ] No REST endpoints introduced
- [ ] PR targets `dev` branch

## Code Style Requirements

- Follow PEP 8 for Python code
- Use type hints throughout the codebase
- Write descriptive docstrings for all functions and classes
- Keep functions small and focused
- Use Ruff for linting and formatting

## Architecture Compliance

### Django + GraphQL Architecture

- Django backend with Strawberry GraphQL integration
- Stateless service design for horizontal scalability
- PostgreSQL for production, SQLite for development/testing
- Redis for caching and message brokering
- Channels for WebSocket/subscription support

### Forbidden Patterns

❌ **These patterns are strictly prohibited:**

- REST API endpoints (Flask routes, Django REST framework views)
- Direct database access bypassing Django ORM
- Blocking synchronous code in async contexts
- Hard-coded configuration values
- Tests that don't enhance coverage
- Documentation outside the `docs/` folder

## Security and Best Practices

- Validate all input data
- Use Django's built-in authentication systems
- Implement proper error handling
- Follow organization-level permissions
- Regular dependency updates
- Comprehensive logging for debugging

---

**Remember**: This repository follows a strict GraphQL-only architecture with comprehensive testing requirements. All changes must target the `dev` branch and enhance test coverage. When in doubt, refer to existing patterns in the codebase and documentation in the `docs/` folder.
